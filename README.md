# Lyra Packaging

Build and packaging infrastructure for [Lyra](https://github.com/lyra-ai-assistant), a local AI assistant for GNU/Linux. This repo produces installable packages for Arch Linux (pacman) and Debian-based distributions (apt).

## Repository structure

```
.
├── artifacts/              # Generated by build-app.sh (not committed)
├── dist/                   # Generated .deb output (not committed)
├── packages/
│   ├── arch/lyra/
│   │   ├── PKGBUILD
│   │   └── lyra.install
│   └── debian/lyra/
│       └── DEBIAN/
│           ├── control
│           └── postinst
└── scripts/
    ├── build-app.sh        # Builds server wheel and UI tarball
    ├── build-arch.sh       # Produces .pkg.tar.zst for pacman
    ├── build-deb.sh        # Produces .deb for apt
    ├── pip_installer.sh    # Shared pip detection helper
    └── deb_builder.py      # Pure Python .deb builder
```

## Prerequisites

| Tool | Required for |
|------|-------------|
| `uv` | Building the Python server |
| `pnpm` | Building the Electron UI |
| `makepkg` | Arch package |
| `python3` | Debian package |
| `uv`, `pip`, or `pip3` | Vendoring Python dependencies |

No `dpkg-deb` required — the `.deb` is assembled by `deb_builder.py` using Python stdlib only.

## Build pipeline

### 1. Build artifacts

```bash
bash scripts/build-app.sh
```

Builds the Python server wheel (`uv build`) and the Electron UI tarball (`pnpm build`), then collects outputs under `artifacts/`.

### 2. Build Arch package

```bash
bash scripts/build-arch.sh
```

Copies artifacts into `packages/arch/lyra/`, runs `makepkg -f`, and outputs `packages/arch/lyra/lyra-<ver>-x86_64.pkg.tar.zst`.

### 3. Build Debian package

```bash
bash scripts/build-deb.sh
```

Stages the full directory tree under `packages/debian/lyra/`, then calls `deb_builder.py` to produce `dist/lyra_<ver>_amd64.deb`.

### Installing

```bash
# Arch
sudo pacman -U packages/arch/lyra/lyra-1.1.0-1-x86_64.pkg.tar.zst

# Debian/Ubuntu
sudo dpkg -i dist/lyra_1.1.0-1_amd64.deb
```

After installation, run the backend setup wizard to download the AI model appropriate for your hardware:

```bash
lyra-install-backend
```

## Design decisions

### Single unified package

The original structure had three separate packages (`lyra-server`, `lyra-ui`, `lyra` metapackage). These were collapsed into a single `lyra` package to simplify installation and avoid dependency resolution between internal packages.

### Vendored Python dependencies

Python dependencies are installed via `pip install --target /usr/lib/lyra/vendor` at build time rather than declared as system package dependencies. This avoids relying on packages that either don't exist in official repos (e.g. `python-chromadb`, `python-huggingface-hub` on Arch) or have different names across distributions. The `lyra` and `lyra-install-backend` wrappers inject this path via `sys.path.insert(0, '/usr/lib/lyra/vendor')`.

### pip installer detection

`scripts/pip_installer.sh` detects the available Python installer in order of preference: `uv` > `pip` > `pip3`. This is sourced by `build-deb.sh` and inlined in the PKGBUILD (makepkg cannot source external files).

### Pure Python .deb builder

`scripts/deb_builder.py` builds valid `.deb` files using Python stdlib only (`tarfile`, `struct`, `io`). A `.deb` is an `ar` archive containing `debian-binary`, `control.tar.gz`, and `data.tar.gz` — no `dpkg-deb` or `binutils` required.

### Artifact separation

`build-app.sh` is responsible only for producing artifacts. Packaging is handled entirely by `build-arch.sh` and `build-deb.sh`. This keeps the build and packaging concerns separate and makes it straightforward to plug in CI.

### Version variables in PKGBUILD

The PKGBUILD uses `_server_ver` and `_ui_ver` variables separate from `pkgver` to handle the fact that the server and UI are versioned independently. `pkgver` represents the release version of the combined `lyra` package.
